Agentic Coding 工作手册

AI Coding 不是少写所有代码，而是减少低价值编码。

成熟的工程师不会把每一次敲键盘都外包给 Agent。真正高效的方式是：人负责判断、边界和验收，Agent 负责搜索、批量执行、重构、测试和跨文件落地。

2秒 已定位、已知道 diff：自己改。
20分 需要查影响面：让 Agent 先研究。
2小时+ 跨模块、重复劳动：让 Agent 主导。

核心原则

判断“谁来改”的关键不是技术炫技，而是谁拥有最高密度的上下文。

谁最清楚，就谁来改。Agent 的价值不是替你按每一个键，而是替你承担搜索、展开、重复和验证。

自己改

你知道文件、函数和 diff

比如 > 改 >=、文案替换、参数调整、SQL 条件微调。让 Agent 接手反而要消耗定位、解释和确认成本。

Agent 改

你知道目标，但不知道路径

比如“报名失败为什么发生”“新增支付渠道要改哪些层”。让 Agent 先读代码、列影响面、给计划，再决定是否执行。

人 Review

Agent 产出必须验收

OpenAI 和 GitHub 都强调：Agent 可以运行测试、提交 PR、展示日志，但人仍要审查和验证生成代码后再合入。

任务特征

自己改

这是低成本手工改动

你已经有准确上下文，直接在编辑器里改最快。不要把 10 秒工作变成 5 分钟对话。

  • 改完后跑相关测试或至少做一次本地检查。
  • 如果改动开始扩散，再切换为 Agent 研究模式。

三种协作模式

不要只有“全交给 AI”和“完全不用 AI”两个档位。高效团队会按任务复杂度切换模式。

Mode 1

手工微改

适合已定位的小改动。工程师直接修改，然后用 Agent 做可选的局部 Review 或测试建议。

  • if 条件、参数、文案、配置
  • 单点 SQL 条件
  • 小范围样式或类型修复
Mode 2

结对探索

适合你知道问题，但还不确定影响面。先让 Agent 只读研究，输出位置、调用链、风险和建议，不急着改。

  • 线上问题定位
  • 陌生模块理解
  • 改动前影响面分析
Mode 3

委派交付

适合跨文件、重复劳动或完整功能。Agent 负责实现和自测，人负责验收标准、产品判断、安全边界和最终 Review。

  • 新增支付渠道、报表、后台能力
  • 批量日志、测试、迁移
  • 大函数拆分和系统性重构

标准工作流

Agent 不是“许愿机”。把任务拆成研究、计划、实现、验证、Review，结果会稳定很多。

1

先写清楚任务边界

说明目标、业务背景、不能改什么、验收标准。GitHub 建议给 Agent 的 issue 要足够清晰且范围适中。

2

先研究，不急着写代码

对于不熟的区域，让 Agent 先定位文件、画调用链、列风险。GitHub 的最佳实践也建议先研究、计划、迭代，再开 PR。

3

给它明确靶子

Anthropic 的经验是：Agent 在有清晰目标时表现更好，例如失败测试、视觉稿、接口契约或明确输出。

4

让 Agent 自己验证

要求它运行相关测试、lint、类型检查，并把失败点说清楚。OpenAI 对 Codex 的说明也强调用户应能查看引用、终端日志和测试结果。

5

人做最终 Review

Agent 写的代码必须像普通同事提交的 PR 一样审查：看行为、测试、安全、可维护性和是否改了不该改的地方。

Prompt 模板

真正有用的 Prompt 不是“帮我改一下”，而是把工程任务交代完整。

研究模式只读，不改代码
请先不要修改代码。

目标：定位活动报名失败的原因。
已知现象：等级为 5 的用户报名失败，等级 6 可以。
范围：活动报名链路，不要分析无关营销模块。

请输出：
1. 相关文件和函数
2. 调用链
3. 最可能的 2 个原因
4. 你建议修改的位置
5. 需要补充的测试用例
实现模式小步修改，可验证
请按下面方案实现，并保持改动最小。

目标：等级 5 及以上用户可以报名活动。
修改范围：活动报名逻辑和相关测试。
禁止：不要改会员等级模型，不要重构无关代码。

验收：
1. Level 4 不可报名
2. Level 5 可以报名
3. Level 6 可以报名
4. 运行相关单元测试并汇报结果

验证与 Review

Agent 做得越多，人越不能只看“它说完成了”。你要看证据。

看 diff

是否改了正确的位置

重点看是否出现“绕路式修复”：改了调用方、加了奇怪兜底、扩大了权限或改了不相关模块。

看测试

是否覆盖边界条件

对 bugfix，至少要有失败用例变绿；对重构，要有行为保持的测试或人工验证路径。

看风险

是否碰到安全和数据

权限、支付、隐私、删除、迁移、密钥和依赖升级，不能只依赖 Agent 自信输出。

团队落地建议

把 Agent 用好，最终是团队工程习惯问题，不是某个工具的 Prompt 技巧。

建立“任务分流”共识

把小改、中改、大改的例子写进团队规范。Review 时也可以提醒：这个改动你自己 10 秒能完成，不必开 Agent 会话。

  • 小改自己来
  • 不确定先让 Agent 研究
  • 跨文件和重复劳动让 Agent 执行

给仓库写 Agent 说明

维护构建命令、测试命令、目录结构、代码风格、禁改区域和 Review 重点。GitHub 和 Codex 类工具都支持通过仓库说明提升 Agent 对项目的理解。

  • 如何启动和测试
  • 模块边界和命名约定
  • 安全、权限、数据库变更规则

让 Agent 并行做低耦合任务

OpenAI 的团队实践提到，Codex 常用于理解复杂系统、重构大代码库、处理新功能和紧急问题。适合拆成多个互不干扰的小任务并行推进。

保留人的产品判断

Agent 很会完成“说清楚的任务”，但不天然理解业务取舍。需求是否正确、风险能否接受、上线节奏如何，仍然是工程师和团队负责。

避免误区

Prompt Driven Development 最大的问题，不是用了 Prompt，而是把清楚的工程判断换成了低效对话。

误区

已定位还让 Agent 搜全项目

这会放大误改概率，也会浪费上下文窗口。直接告诉它文件和行，或自己改。

误区

一句“优化一下”就开工

没有目标、验收和边界，Agent 只能猜。猜对是运气，猜错是常态。

误区

只看总结，不看证据

不要因为 Agent 写了“已完成”就合入。看 diff、日志、测试、风险说明。
