Agentic Coding 工作手册

AI Coding 不是少写所有代码，而是减少低价值编码。

成熟的工程师不会把每一次敲键盘都外包给 Agent。真正高效的方式是：人负责判断、边界和验收，Agent 负责搜索、批量执行、重构、测试和跨文件落地。

2秒 已定位、已知道 diff：自己改。
20分 需要查影响面：让 Agent 先研究。
2小时+ 跨模块、重复劳动：让 Agent 主导。

核心原则

判断“谁来改”的关键不是技术炫技，而是谁拥有最高密度的上下文。

谁最清楚，就谁来改。Agent 的价值不是替你按每一个键，而是替你承担搜索、展开、重复和验证。

自己改

你知道文件、函数和 diff

比如 > 改 >=、文案替换、参数调整、SQL 条件微调。让 Agent 接手反而要消耗定位、解释和确认成本。

Agent 改

你知道目标，但不知道路径

比如“报名失败为什么发生”“新增支付渠道要改哪些层”。让 Agent 先读代码、列影响面、给计划，再决定是否执行。

人 Review

Agent 产出必须验收

OpenAI 和 GitHub 都强调：Agent 可以运行测试、提交 PR、展示日志，但人仍要审查和验证生成代码后再合入。

任务特征

自己改

这是低成本手工改动

你已经有准确上下文，直接在编辑器里改最快。不要把 10 秒工作变成 5 分钟对话。

  • 改完后跑相关测试或至少做一次本地检查。
  • 如果改动开始扩散，再切换为 Agent 研究模式。

三种协作模式

不要只有“全交给 AI”和“完全不用 AI”两个档位。高效团队会按任务复杂度切换模式。

Mode 1

手工微改

适合已定位的小改动。工程师直接修改，然后用 Agent 做可选的局部 Review 或测试建议。

  • if 条件、参数、文案、配置
  • 单点 SQL 条件
  • 小范围样式或类型修复
Mode 2

结对探索

适合你知道问题，但还不确定影响面。先让 Agent 只读研究，输出位置、调用链、风险和建议，不急着改。

  • 线上问题定位
  • 陌生模块理解
  • 改动前影响面分析
Mode 3

委派交付

适合跨文件、重复劳动或完整功能。Agent 负责实现和自测，人负责验收标准、产品判断、安全边界和最终 Review。

  • 新增支付渠道、报表、后台能力
  • 批量日志、测试、迁移
  • 大函数拆分和系统性重构

标准工作流

Agent 不是“许愿机”。把任务拆成研究、计划、实现、验证、Review，结果会稳定很多。

1

先写清楚任务边界

说明目标、业务背景、不能改什么、验收标准。GitHub 建议给 Agent 的 issue 要足够清晰且范围适中。

2

先研究，不急着写代码

对于不熟的区域，让 Agent 先定位文件、画调用链、列风险。GitHub 的最佳实践也建议先研究、计划、迭代，再开 PR。

3

给它明确靶子

Anthropic 的经验是：Agent 在有清晰目标时表现更好，例如失败测试、视觉稿、接口契约或明确输出。

4

让 Agent 自己验证

要求它运行相关测试、lint、类型检查，并把失败点说清楚。OpenAI 对 Codex 的说明也强调用户应能查看引用、终端日志和测试结果。

5

人做最终 Review

Agent 写的代码必须像普通同事提交的 PR 一样审查：看行为、测试、安全、可维护性和是否改了不该改的地方。

Prompt 模板

真正有用的 Prompt 不是“帮我改一下”，而是把工程任务交代完整。

研究模式只读，不改代码
请先不要修改代码。

目标：定位活动报名失败的原因。
已知现象：等级为 5 的用户报名失败，等级 6 可以。
范围：活动报名链路，不要分析无关营销模块。

请输出：
1. 相关文件和函数
2. 调用链
3. 最可能的 2 个原因
4. 你建议修改的位置
5. 需要补充的测试用例
实现模式小步修改，可验证
请按下面方案实现，并保持改动最小。

目标：等级 5 及以上用户可以报名活动。
修改范围：活动报名逻辑和相关测试。
禁止：不要改会员等级模型，不要重构无关代码。

验收：
1. Level 4 不可报名
2. Level 5 可以报名
3. Level 6 可以报名
4. 运行相关单元测试并汇报结果

验证与 Review

Agent 做得越多，人越不能只看“它说完成了”。你要看证据。

看 diff

是否改了正确的位置

重点看是否出现“绕路式修复”：改了调用方、加了奇怪兜底、扩大了权限或改了不相关模块。

看测试

是否覆盖边界条件

对 bugfix，至少要有失败用例变绿；对重构，要有行为保持的测试或人工验证路径。

看风险

是否碰到安全和数据

权限、支付、隐私、删除、迁移、密钥和依赖升级，不能只依赖 Agent 自信输出。

真实案例：从需求改动到上线

下面用一个常见业务改动串完整流程：什么时候自己改，什么时候让 AI 研究、实现、补测试和准备上线材料。

业务背景

活动报名等级规则调整

运营反馈：原规则是“用户等级大于 5 才能报名活动”，现在活动策略改为“用户等级大于等于 5 即可报名”。表面看只是 > 改成 >=，但它可能同时影响活动详情页、报名接口、后台配置、风控提示、测试用例和上线验证。

目标

不是让 AI 代替你改等号

如果你已经知道就在 JoinActivity() 里改一行，那就自己改。但在真实项目里，程序员更常面对的是：规则散落在哪里、是否有前端文案、是否有缓存、是否有测试、上线后怎么证明生效。AI 的价值就在这些环节。

  • 人负责判断业务语义、风险和上线决策。
  • AI 负责读代码库、找影响面、补测试、生成检查清单。
  • 小 diff 人直接改，大范围改动让 AI 执行，人 Review。
阶段 人主导 AI 辅助 输出物
需求澄清 确认“等级 5 可报名”的业务含义、活动范围、是否只影响新活动。 把需求改写成验收标准、列边界问题。 验收清单：Level 4 拒绝，Level 5/6 通过。
影响面分析 判断哪些模块是核心链路，哪些不能乱改。 搜索报名链路、等级判断、错误文案、测试和后台配置。 文件列表、调用链、风险点。
代码修改 如果只是一行已定位代码，自己改。 如果散落多处，让 AI 按计划小步修改。 最小 diff，不做无关重构。
测试验证 判断测试是否覆盖真实业务边界。 补单测、接口用例、回归清单，运行测试并解释失败。 测试日志、失败修复、手工验证步骤。
上线准备 决定灰度范围、回滚策略、是否通知运营和客服。 生成 PR 描述、发布说明、上线检查表、监控关注项。 PR、发布单、回滚说明。
上线后 确认指标正常，判断是否扩大灰度或全量。 整理日志查询语句、监控项、复盘记录。 上线验证截图、监控结果、经验沉淀。
第 0 步接需求

先把一句需求变成可验收规则

不要直接开改。先确认“等级 5”是用户当前等级、历史最高等级，还是活动报名时冻结的等级；确认老活动是否受影响。

人做 问产品和运营：范围、边界、上线时间、失败提示是否要变。
AI 做 把需求整理成验收标准和待确认问题，避免漏问。
第 1 步查影响面

不知道散落在哪，就让 AI 先只读研究

真实项目里同一个规则可能出现在服务端校验、前端按钮置灰、错误码文案、后台预览、数据报表里。此时不要靠记忆硬改。

人做 告诉 AI 搜索范围：活动报名链路优先，营销报表暂不改。
AI 做 搜索 Level > 5、报名错误码、活动测试，输出调用链和候选修改点。
第 2 步决定谁改

已定位的一行，自己改；多文件一致性，让 AI 改

如果 AI 研究后发现只有一个函数判断错了，程序员直接敲 =。如果同时要改接口测试、前端提示、后台预览，就让 AI 按清单执行。

人做 选择策略：手改核心一行，或让 AI 批量同步相关文件。
AI 做 按“最小改动”原则修改，不顺手重构、不扩大会员体系设计。
第 3 步补测试

测试用 AI 很划算，因为边界样例重复但重要

这个需求至少要覆盖 Level 4、5、6。若有活动开关、库存、报名时间，还要保证新规则没有掩盖其他失败原因。

人做 确认哪些业务边界必须测，哪些只是已有用例复用。
AI 做 补单测和接口测试，运行相关命令，失败时先解释原因再修。
第 4 步提 PR

让 AI 写材料，人检查结论

PR 描述、风险点、测试结果、上线影响、回滚方式都适合让 AI 起草。程序员要检查它有没有夸大覆盖范围，或者漏掉手工验证。

人做 Review diff：是否只改活动报名，是否出现无关重构。
AI 做 生成 PR 摘要、测试证据、发布说明和 Reviewer 关注点。
第 5 步灰度上线

上线不是“代码合了就结束”

先用内部账号或小流量验证 Level 5 可报名，同时观察 Level 4 拦截是否正常、报名失败率是否异常、客服反馈是否变少。

人做 决定灰度比例、上线窗口、回滚触发条件。
AI 做 准备验证步骤、日志查询语句、监控看板说明和回滚清单。
第 6 步复盘沉淀

把这次经验变成下一次 Agent 更懂项目的上下文

如果发现等级规则散落多处，就补团队规范：等级判断统一走一个函数；如果发现测试缺边界，就补测试模板。

人做 决定是否抽公共能力、补规范、补监控。
AI 做 整理复盘、生成代码库 Agent 说明、提出下次可自动化的检查项。

这条链路里的关键不是“让 AI 改不改等号”，而是让程序员知道：需求澄清、影响面分析、测试、发布和复盘，每一步都有不同的人机分工。

团队落地建议

把 Agent 用好，最终是团队工程习惯问题，不是某个工具的 Prompt 技巧。

建立“任务分流”共识

把小改、中改、大改的例子写进团队规范。Review 时也可以提醒：这个改动你自己 10 秒能完成，不必开 Agent 会话。

  • 小改自己来
  • 不确定先让 Agent 研究
  • 跨文件和重复劳动让 Agent 执行

给仓库写 Agent 说明

维护构建命令、测试命令、目录结构、代码风格、禁改区域和 Review 重点。GitHub 和 Codex 类工具都支持通过仓库说明提升 Agent 对项目的理解。

  • 如何启动和测试
  • 模块边界和命名约定
  • 安全、权限、数据库变更规则

让 Agent 并行做低耦合任务

OpenAI 的团队实践提到，Codex 常用于理解复杂系统、重构大代码库、处理新功能和紧急问题。适合拆成多个互不干扰的小任务并行推进。

保留人的产品判断

Agent 很会完成“说清楚的任务”，但不天然理解业务取舍。需求是否正确、风险能否接受、上线节奏如何，仍然是工程师和团队负责。

避免误区

Prompt Driven Development 最大的问题，不是用了 Prompt，而是把清楚的工程判断换成了低效对话。

误区

已定位还让 Agent 搜全项目

这会放大误改概率，也会浪费上下文窗口。直接告诉它文件和行，或自己改。

误区

一句“优化一下”就开工

没有目标、验收和边界，Agent 只能猜。猜对是运气，猜错是常态。

误区

只看总结，不看证据

不要因为 Agent 写了“已完成”就合入。看 diff、日志、测试、风险说明。
