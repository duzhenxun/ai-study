THE STARTING POINT

而是先问：想证明什么？

同一个活动，“快速可用”“完整回归”“奖池专项”“沉淀框架”需要完全不同的测试深度和数据成本。

  • 03专项能力验证

  • 05其他可验证目标

07 / 09
FROM FIXED FLOW TO CAPABILITY DISCOVERY

Skill 不应该记住某个活动，
而应该学会读懂活动

早期写法：固定业务流程

  • 页面 → 兑换 → 抽奖 → 记录
  • 默认所有活动都有奖池
  • 数据不足就只能阻塞
  • 测试点靠人工提前给出
→

最终写法：动态能力发现

  • 读取 Go route / JSON / Wiki
  • 识别任务、榜单、报名、兑换、领奖等能力
  • 为每种能力生成数据与边界矩阵
  • 选择最低成本主动作和结果面
抽奖只是能力类型之一Mechanism over business type
inke-act-test 最终结构
08 / 09
THE REUSABLE PIPELINE

活动名进来，动态测试矩阵出去

01 · INPUTmodule + env + UID + Wiki

不再要求前端目录和 H5 地址

02 · DISCOVERGo / JSON / H5 / API

自动匹配 ali-testqa 或 ali-gray

03 · GOAL GATE先明确最想证明什么

冒烟、回归、专项、框架

04 · CAPABILITY发现活动能力

任务、榜单、报名、兑换、领奖……

05 · DATA数据准备与造数 dry-run

READY / NEEDS_SEED / BLOCKED

06 · BOUNDARY边界与异常 case

时间、数值、资格、库存、幂等

07 · EXECUTE最小 mutation 执行

before / action / after

08 · DELIVER报告、CI 与审计证据

产品失败与环境阻塞分开

时间与 version 是测试前置门禁Reusable by design
周会结论
09 / 09
TAKEAWAYS

这次真正沉淀的，
不是一份报告，而是测试决策方式

01

先问目标，再决定测试深度。

02

先发现能力，再生成测试矩阵。

03

每个能力都要配数据准备和边界 case。

04

结论必须由 UI、API 和状态证据共同支持。

05

产品失败、数据阻塞、权限阻塞必须分开。

06

Skill 抽象机制，不写死某一种业务。
