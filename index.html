第二部分 Agent 开发入门 / s01

最小 Agent Loop：让模型持续行动

Agent 不是一次问答，而是「模型思考 → 调工具 → 写回结果 → 再思考」的循环。

要解决的问题：大模型只会生成下一段文本；没有宿主程序，它不会读文件、跑命令，也不知道工具执行结果。
核心机制：messages 保存对话状态；模型输出 tool_use；程序执行工具；tool_result 作为新消息回填。
可以这样理解：Agent 的「行动能力」其实来自循环，不是来自某一次神奇回答。
工程重点：循环要可追踪、可停止、可恢复；每一步都要保留是谁请求、执行了什么、返回了什么。
最小闭环：四步循环起来才叫 Agent
1用户目标把任务和历史消息放进 messages
→
2模型决策回复文本，或声明需要调用哪个工具
↑
↓
4结果回填tool_result 回到 messages，进入下一轮
←
3宿主执行本地程序真正读文件、跑命令、访问 API

模型负责「决定下一步」，程序负责「真实执行」，上下文负责「让下一轮知道发生过什么」。

第一步是跑通闭环，不是堆功能10
第二部分 Agent 开发入门 / s02

添加工具：把能力挂到 Agent 上

工具是 Agent 和真实世界之间的接口，必须同时给模型「说明书」和给程序「执行函数」。

要解决的问题：只有 bash 时, 所有操作都走 shell。cat 截断不可预测, sed 遇到特殊字符就崩, 每次 bash 调用都是不受约束的安全面。
核心机制：schema 告诉模型工具名、参数和用途；handler 在宿主侧接收参数并返回结果。
API 协议有三条硬性约束:
- 每个 tool_use 块必须有匹配的 tool_result (通过 tool_use_id 关联)
- user / assistant 消息必须严格交替 (不能连续两条同角色)
- 只接受协议定义的字段 (内部元数据会导致 400 错误)

工具不是越多越好，关键是每个工具的用途、输入、风险和返回格式都足够清楚。

工具 = schema + handler + 结果回写11
第二部分 Agent 开发入门 / s03

Todo 管理：让 Agent 有当前工作记忆

Todo 不是给用户看的装饰，而是给 Agent 自己用的短期工作台。

要解决的问题：到了 s02，agent 已经会读文件、写文件、跑命令。 问题也马上出现了：
多步任务容易走一步忘一步。
明明已经做过的检查，会重复再做。
一口气列出很多步骤后，很快又回到即兴发挥
核心机制：把计划拆成 pending、in_progress、completed 三类状态，执行过程中持续更新。同一时间只允许一个 in_progress；完成后要更新状态；计划变化要同步调整而不是硬撑旧计划。
可以这样理解：Todo 像施工看板，让 Agent 每次行动前都能看见「当前唯一焦点」。
Todo 看板：把复杂任务变成可检查状态

第二部分 Agent 开发入门 / s04

子代理：把局部探索放到独立上下文

一个大任务，不一定要塞进一个上下文里做完。

要解决的问题：当 agent 连续做很多事时，messages 会越来越长。比如用户只问：“这个项目用什么测试框架？”
但 agent 可能为了回答这个问题 开始扫仓库、查资料会产生大量中间信息，直接塞进主上下文会很快变乱。
真正有价值的最终答案，可能只有一句话：“这个项目主要是技术分享。”
核心机制： 当前正在和用户对话的agent就是父智能体。父智能体临时派生出来，专门处理某个子任务的 agent，就是子智能体。子智能体也有自己的 messages，不会和父智能体的 messages 混淆。
可以这样理解：把局部任务放进独立上下文里做，做完只把必要结果带回来。
子智能体通常不需要拥有和父智能体完全一样的能力。

父智能体除了文件读取、搜索、bash 之类的基础工具。它还有派生子智能体的能力

并行不是目的，隔离噪音才是目的13
第二部分 Agent 开发入门 / s05

Skill 加载：按需读取能力说明

Skill 是可发现、可按需加载的能力包；常驻的是目录，不是全部正文。

要解决的问题：
不同任务需要的领域知识不一样
写日报、周报
写活动需求文档
做PPT演示文稿
核心机制：启动时扫描 name 和 description；把 skill 目录放进 system prompt；提供一个load_skill 函数，当任务命中特定能力，load_skill再读取对应 SKILL.md、脚本和资源。
skill、memory、CLAUDE.md|AGENT.md 的边界

这是某类任务才需要的做法或知识：skill

这是需要长期记住的事实或偏好：memory

这是更稳定的全局规则：CLAUDE.md|AGENT.md

Skill 加载：目录常驻，正文按需打开

常驻目录

ppt-outline做 PPT 大纲和 HTML 演示文稿
wiki-skill读写公司 Wiki 页面
playwright-cli自动化浏览器验证

命中后加载

SKILL.md完整流程、判断规则、输出格式
scripts/确定性脚本，减少模型自由发挥

好 Skill 的第一门槛是「能被正确触发」，第二门槛才是「触发后能稳定执行」。

目录常驻，正文按需加载14
第二部分 Agent 开发入门 / s06

上下文压缩：保留状态，不保留噪音

压缩不是简单变短，而是把后续继续工作必需的状态留下来。

要解决的问题：前面讲到了s05，agent 已经会：读写文件、规划步骤、派子 agent、按需加载 skill。 也正因为它会做的事情更多了，上下文会越来越快膨胀读。一个大文件，会塞进很多文本。跑一条长命令，会得到大段输出。多轮任务推进后，旧结果会越来越多。API 请求越来越重，越来越贵......
核心机制： 上下文压缩的核心，不是尽量少字，而是让模型在更短的活跃上下文里，仍然保住继续工作的连续性。
先记三层，不要一上来记八层十层

第 1 层：大结果不直接塞进上下文。写到磁盘，只留预览

第 2 层：旧结果不一直原样保留。替换成简短占位，used tool with id：tool_call_id

第 3 层：整体历史太长时。生成一份连续性摘要

压缩策略：原文可追溯，当前状态可继续

手动压缩和自动压缩复用同一条机制

压缩不是“把历史缩短”这么简单15
第二部分 Agent 开发入门 / s07

权限系统：工具调用先过安全闸

“意图”不能直接变成“执行”，中间必须经过权限检查。

要解决的问题：模型可能会写错文件，模型可能会执行危险命令，模型可能会在不该动手的时候动手
核心机制：模型可以提出行动建议，但真正执行之前，必须先过安全关。
简单讲4步

第 1 步先看 deny rules。因为有些东西不应该交给“模式”去决定。比如：明显危险的命令、明显越界的路径 这些应该优先挡掉。

第 2 步看 mode。因为模式决定当前会话的大方向。例如在 plan 模式下，系统就应该天然更保守。

第 3 步看 allow rules。有些安全、重复、常见的操作可以直接过。比如：读文件、搜索代码、查看 git 状态

第 4 步才 ask。前面都没命中的灰区，才交给用户。

最小实现怎么写

权限判断属于系统责任，不能让模型自己决定「这次应该安全」。

越能干的 Agent 越需要权限边界16
第二部分 Agent 开发入门 / s08

Hook 系统：在关键生命周期插入扩展

Hook 把日志、审计、格式化、通知等横切能力放到生命周期节点上，而不是塞进主循环。

要解决的问题：在某个固定时机顺手做一点事，不改主循环主体，也能接入额外规则，让用户或插件在系统边缘扩展能力。
比如 会话开始时打印欢迎信息、工具执行前做一次额外检查、工具执行后补一条审计日志
核心机制：在 SessionStart、PreToolUse、PostToolUse、Stop 等阶段触发 hook。
可以这样理解：Hook 像流程里的插座：核心电路不变，需要扩展时把插件接上去。
生命周期 Hook：扩展挂在节点上

Hook 的价值是让核心循环保持干净，同时给团队留下稳定的扩展点。

Hook 让核心循环保持干净17
第二部分 Agent 开发入门 / s09

记忆系统：只记未来仍有价值的东西

记忆不是聊天记录仓库，而是把未来还会用到的事实、偏好和项目规则保存下来。

要解决的问题：如果一个 agent 每次新会话都完全从零开始，它就会不断重复忘记这些事情，会忘记项目怎么跑、用户偏好、团队约定和上次验证过的结论。
核心机制：不只保存“你做错了”，也要保存“这样做是对的”。有些东西即使用户要求你存，也不该直接存。用户说“忽略 memory”时，就当它是空的。在这一轮里，按 memory 为空来工作
哪些东西不要存进 memory ：

文件结构、函数签名、目录布局

当前任务进度

临时分支名、当前 PR 号

修 bug 的具体代码细节

密钥、密码、凭证

memory 保存的是“以后还可能有价值、但当前代码里不容易直接重新看出来”的信息。
用户偏好回答希望简洁还是详细、更偏好什么工具链、明确说过要记住的默认值。
项目规则测试命令、目录约定、代码风格、部署入口。下次任务仍然有用。
明确纠正过不要这样改、这个判断方式之前错过、以后遇到这种情况要先做 X
外部资源指针某个问题单在哪个看板里、某个监控面板在哪里、某个资料库在哪个 URL

好记忆要能回答「下次遇到同类任务，它能少走哪一步弯路？」

第二部分 Agent 开发入门 / s10

系统提示词：用拼装流水线管理规则

Prompt 不是一整坨文本，而是由多层规则按优先级拼装出来的运行配置。

要解决的问题：系统提示词会越来越长：平台规则、项目规则、工具说明、Skill、当前任务约束混在一起难维护。
核心机制：按来源拼装：基础行为规则、项目指令、工具目录、Skill 摘要、权限模式、当前任务 brief。
可以这样理解：Prompt 像应用启动配置，不应该手写一份巨大字符串，而应该有稳定的装配顺序。
工程重点：冲突要有优先级；动态内容不要污染长期规则；每一层来源都要可追踪。
Prompt 组装：稳定顺序 + 清晰来源
1
基础 Agent 行为规则
最高优先级
2
项目 AGENTS.md / 团队约束
本仓库规则
3
工具目录 / Skill 摘要 / 权限模式
运行时能力
4
当前任务目标、计划、已知约束
本轮动态

讲解重点：提示词工程到工程化阶段，核心不是写一句神奇 prompt，而是管理规则来源和优先级。

Prompt 是运行配置，不是一坨文本19
第二部分 Agent 开发入门 / s11

错误恢复：失败要能被理解和重试

好 Agent 不是不失败，而是能把失败转成可理解的状态，并选择下一步恢复动作。

要解决的问题：命令失败、路径不存在、网络超时、权限拒绝都会打断任务；整屏 stderr 会让模型更混乱。
核心机制：把错误分类成可重试、需改参数、需用户输入、不可恢复，并给出下一步建议。防止无限重试，让每种恢复路径各算各的次数
可以这样理解：错误恢复像客服分诊：先判断病因，再决定重试、换方案、问用户，还是明确停止。
工程重点：返回短摘要、退出码、关键错误片段、已尝试动作和建议动作；避免把噪音原封不动塞回上下文。
好 Agent 不是不失败，而是会恢复20
第二部分 Agent 开发入门 / s12

任务系统：把大目标拆成持久工作图

Todo 管当前注意力，任务系统管跨会话、跨角色、可恢复的长期工作。

要解决的问题： 不是单纯的TODO列表，而是一张“依赖关系图”。
如果没有专门的任务系统，agent 很容易出现这些问题：
前置工作没做完，就贸然开始后面的任务
某个任务完成以后，不知道解锁了谁
多个 agent 协作时，没有统一任务板可读
所以这一章要做的升级是：
把“会话里的 todo”升级成“可持久化的任务图”。
核心机制：把大目标拆成 task graph；每个 task 有 pending、ready、running、blocked、done 等状态。
可以这样理解：后端与设计同学，需等产品同学完成后才能开始。前端研发同学必须等设计同学完成，才能开始开发。测试同学必须等研发同学完成，才能开始测试。
任务落到目录文件 .tasks/task_xxx.json

最小工具集：task_create、task_update、task_get、task_list

todo 更像本轮计划，task 更像长期工作板/span>21
第二部分 Agent 开发入门 / s13

后台任务：慢操作不阻塞主循环

后台任务让慢操作脱离主循环等待，但结果仍要回到同一套任务和消息系统。

要解决的问题：跑测试、构建、爬数据、批量处理都可能很久；如果同步等待，Agent 主循环会被卡住。
核心机制：background_run 立即返回 task_id；慢执行写日志；完成后通过通知队列把摘要送回主循环。
可以这样理解：不是复制一个主脑并行思考，而是把耗时动作放到后台执行槽位。
工程重点：后台任务要能查询状态、取消、超时；通知只放摘要，完整输出放文件或 artifact。
后台执行流：快返回，慢执行，完成再通知

讲解重点：后台任务不是脱离系统单独跑，最终结果必须回到任务状态和上下文里。

并行的是执行槽位，不是复制主脑22
第二部分 Agent 开发入门 / s14

定时调度：把未来任务放回 Agent Loop

调度器只负责在未来唤醒 Agent，真正思考和执行仍然进入 Agent Loop。

要解决的问题：提醒、巡检、日报、周报、监控回访不能依赖当前会话一直开着。
核心机制：保存 schedule + prompt；到时间后把 prompt 作为新输入追加回消息系统。
可以这样理解：定时器像闹钟，它只负责叫醒；醒来之后怎么做，仍由 Agent 根据上下文、权限和工具完成。
工程重点：触发要可追踪、可暂停、可更新；执行结果仍走权限、工具、任务状态和错误恢复。
调度唤醒：未来输入回到同一个 Loop
创建计划每天 09:00 / 每周一 / 30 分钟后
→
保存 prompt任务内容自包含，可独立唤醒
到点触发scheduler due
→
追加新输入进入 messages / task queue
Agent Loop 执行权限、工具、记忆、错误恢复
→
产出结果通知用户或更新任务

调度能力的关键是「回到统一运行时」，而不是在定时器里另写一套逻辑。

定时器只负责唤醒，不负责思考23
第二部分 Agent 开发入门 / s15

Agent 团队：让角色长期存在

多 Agent 不是一次性 subagent，而是让不同角色长期存在、拥有职责和状态。

要解决的问题：s04 的 subagent 已经能帮主 agent 拆小任务。但 subagent 有一个很明显的边界：创建 -> 执行 -> 返回摘要 -> 消失。这很适合一次性的小委派。
可如果你想做这些事，就不够用了：
让两个 agent 长期分工
让某个 agent 未来收到新任务后继续工作
也就是说，系统现在缺的不是“再开一个模型调用”，而是：一批有身份、能长期存在、能反复协作的队友。
实现核心：

1、先有一份队伍名册

2、spawn 一个持久队友

3、给每个队友一个邮箱，最简单的做法可以直接用 JSONL 文件

4、队友每轮先看邮箱，再继续工作

团队系统至少需要“名册 + 邮箱 + 独立循环”

每个队友都应该有自己的 messages 和自己的 inbox。

队友不是一次性函数调用24
第二部分 Agent 开发入门 / s16

团队协议：协作要有可追踪规则

有了邮箱以后，团队已经能说话；有了协议以后，团队才开始会“按规矩协作”。

要解决的问题：s15 已经让队友之间可以互相发消息。 但如果所有事情都只靠自由文本，会有两个明显问题： 某些动作必须明确批准或拒绝，不能只靠一句模糊回复。
一旦多个请求同时存在，系统很难知道“这条回复对应哪一件事” 最典型的两个场景就是：
队友要不要优雅关机
某个高风险计划要不要先审批
核心机制：任务认领、状态更新、消息、交接、退出都走结构化协议。双方提前约定好“消息长什么样、收到以后怎么处理”。
可以这样理解：协议像团队协作制度：谁认领、谁负责、什么时候交接，都要写到系统里。
团队协议：把协作动作结构化

没有协议的多 Agent 只是多条聊天；有协议才是可追踪的协作系统。

多 Agent 需要协议，不只需要热闹25
第二部分 Agent 开发入门 / s17

自主代理：空闲时自己找活干

自治不是随便发挥，而是在任务板和协议约束下，从 idle 状态主动认领 ready task。

要解决的问题：长期团队不能靠 Lead 每一步手动叫人；空闲角色应该能自己发现可做工作。
核心机制：IDLE 时轮询 inbox 和 task board；发现 ready task 后原子认领，进入 working。
可以这样理解：自主代理像有规则的值班同事：先看有没有消息，再看有没有可认领任务。
工程重点：认领要加锁，避免重复工作；没有 ready task 就保持 idle，不要自己编任务。
自治循环：只在规则下主动找活

自治的边界来自任务系统和团队协议，不是让 Agent 无限自由发挥。

自治是在规则下认领 ready task26
第二部分 Agent 开发入门 / s18

Worktree 隔离：队友各自在独立工作区动手

多 Agent 真正同时改代码时，需要用 worktree / branch 把工作区隔离开。

要解决的问题：多个 Agent 在同一个目录里改文件，很容易覆盖、误删、混入彼此未完成的改动。
核心机制：每个任务或角色使用独立 worktree 和 branch；完成后通过 diff、测试、合并流程汇总。
可以这样理解：Worktree 像给每个队友一条独立车道，大家并行开车，最后在合并口检查汇入。
工程重点：分配写入范围；合并前看 diff 和测试；遇到冲突人工或主代理裁决，不自动吞掉别人的修改。
Worktree 车道：并行动手，集中汇总

Worktree 降低冲突概率，但不能代替合并前的测试和代码审查。

隔离降低冲突，汇总需要校验27
第二部分 Agent 开发入门 / s19

MCP与插件：把外部能力标准化接入

工具不必都写死在主程序里。外部进程也可以把能力接进你的 agent。

要解决的问题：能不能让外部程序也把工具接进来，而不用每次都改主程序？
核心机制：

1. 有一个 MCP client、2. 有一个工具名前缀规则、3. 有一个统一路由器

Plugin、MCP Server、MCP Client 不要混成一层

MCP解决的是外部工具怎么通信，plugin 解决的是这些外部工具配置怎么被发现。

平台化以后，能力来自生态28
