第二部分 Agent 开发入门 / s01

最小 Agent Loop：让模型持续行动

Agent 不是一次问答，而是「模型思考 → 调工具 → 写回结果 → 再思考」的循环。

要解决的问题：大模型只会生成下一段文本；没有宿主程序，它不会读文件、跑命令，也不知道工具执行结果。
核心机制：messages 保存对话状态；模型输出 tool_use；程序执行工具；tool_result 作为新消息回填。
可以这样理解：Agent 的「行动能力」其实来自循环，不是来自某一次神奇回答。
工程重点：循环要可追踪、可停止、可恢复；每一步都要保留是谁请求、执行了什么、返回了什么。
最小闭环：四步循环起来才叫 Agent
1用户目标把任务和历史消息放进 messages
→
2模型决策回复文本，或声明需要调用哪个工具
↑
↓
4结果回填tool_result 回到 messages，进入下一轮
←
3宿主执行本地程序真正读文件、跑命令、访问 API

模型负责「决定下一步」，程序负责「真实执行」，上下文负责「让下一轮知道发生过什么」。

第一步是跑通闭环，不是堆功能10
第二部分 Agent 开发入门 / s02

添加工具：把能力挂到 Agent 上

工具是 Agent 和真实世界之间的接口，必须同时给模型「说明书」和给程序「执行函数」。

要解决的问题：没有工具，Agent 只能聊；有了 read_file、write_file、bash、search，它才开始能做事。
核心机制：schema 告诉模型工具名、参数和用途；handler 在宿主侧接收参数并返回结果。
可以这样理解：schema 像菜单，handler 像后厨。模型点菜，程序做菜，结果再端回上下文。
工程重点：强工具必须有边界：校验路径、限制命令、设置超时、截断大输出、把错误包装成模型能理解的信息。
工具路由：模型看 schema，程序跑 handler
read_file 读项目文件
write_file 修改文件
bash 运行命令
search 查外部资料
模型选择工具→参数按 schema 填写
权限和校验→handler 真实执行
工具返回结果→写回 messages

讲解重点：工具不是越多越好，关键是每个工具的用途、输入、风险和返回格式都足够清楚。

工具 = schema + handler + 结果回写11
第二部分 Agent 开发入门 / s03

Todo 管理：让 Agent 有当前工作记忆

Todo 不是给用户看的装饰，而是给 Agent 自己用的短期工作台。

要解决的问题：任务一长，模型容易忘记已经做过什么、下一步该做什么、哪些检查还没完成。
核心机制：把计划拆成 pending、in_progress、completed 三类状态，执行过程中持续更新。
可以这样理解：Todo 像施工看板，让 Agent 每次行动前都能看见「当前唯一焦点」。
工程重点：同一时间只允许一个 in_progress；完成后要更新状态；计划变化要同步调整而不是硬撑旧计划。
Todo 看板：把复杂任务变成可检查状态

第二部分 Agent 开发入门 / s04

子代理：把局部探索放到独立上下文

子代理用独立上下文做局部任务，主代理保留决策权和最终整合权。

要解决的问题：扫仓库、查资料、跑验证会产生大量中间信息，直接塞进主上下文会很快变乱。
核心机制：主 Agent 发出清晰任务；子 Agent 在自己的上下文里探索；最后只返回摘要、证据和建议。
可以这样理解：主代理像负责人，子代理像临时调研员；调研员带回结论，不把全部草稿搬进会议室。
工程重点：任务边界要窄，输入资料要少，输出格式要固定；不要把子会话原文整段回填。
上下文隔离：探索在旁路，决策在主线

主 Agent

保存用户目标、全局约束、最终方案。

任务拆分
派发子任务
整合结论并执行

子 Agent

只处理一个局部问题，用完即返回。

查认证代码在哪里
收集文件和行号证据
返回：摘要 + 证据 + 风险

讲解重点：并行不是目的，隔离噪音才是目的；主代理需要的是可信结论，而不是更多上下文。

子代理负责探索，主代理负责决策13
第二部分 Agent 开发入门 / s05

Skill 加载：按需读取能力说明

Skill 是可发现、可按需加载的能力包；常驻的是目录，不是全部正文。

要解决的问题：几十个 Skill 全塞进系统提示词会浪费上下文，也会让模型被无关规则干扰。
核心机制：启动时扫描 name 和 description；当任务命中特定能力，再读取对应 SKILL.md、脚本和资源。
可以这样理解：Skill 像工具书目录：先看目录决定要不要翻书，真正需要时再打开那一章。
工程重点：description 要写清触发场景；SKILL.md 要给步骤、边界、示例、验证方式；引用文件路径必须可靠。
Skill 加载：目录常驻，正文按需打开

常驻目录

ppt-outline做 PPT 大纲和 HTML 演示文稿
wiki-skill读写公司 Wiki 页面
playwright-cli自动化浏览器验证

命中后加载

SKILL.md完整流程、判断规则、输出格式
scripts/确定性脚本，减少模型自由发挥

好 Skill 的第一门槛是「能被正确触发」，第二门槛才是「触发后能稳定执行」。

目录常驻，正文按需加载14
第二部分 Agent 开发入门 / s06

上下文压缩：保留状态，不保留噪音

压缩不是简单变短，而是把后续继续工作必需的状态留下来。

要解决的问题：长对话、搜索结果、日志和代码输出会挤占上下文，导致 Agent 忘记目标或丢掉关键约束。
核心机制：原始记录落盘；上下文里留下摘要、当前目标、已做决策和下一步。
可以这样理解：压缩像会议纪要：不复印所有聊天记录，只保留继续推进项目所需的信息。
工程重点：保留可追溯性；明确哪些内容可丢、哪些必须留；压缩后 Agent 应该能无缝接着干。
压缩策略：原文可追溯，当前状态可继续
原始记录完整对话、工具输出、日志
落盘保存transcript
必要时回查保证证据不丢
保留状态继续工作所需
目标 / 约束用户要什么，不能做什么
决策 / 下一步已经决定什么，接下来做什么
丢弃噪音不再逐字保留
重复输出长日志、重复搜索片段
中间草稿已被结论覆盖的信息

压缩成功的标准不是 token 变少，而是换一个 Agent 读完摘要也能继续完成任务。

压缩的目标是可继续工作15
第二部分 Agent 开发入门 / s07

权限系统：工具调用先过安全闸

工具越强，越不能让模型直接越过安全边界；权限判断必须在执行前发生。

要解决的问题：删除文件、执行命令、联网请求、写入外部系统都可能造成真实损失。
核心机制：每个工具调用先进入 policy check，再得到 allow、ask user、deny 三种结果。
可以这样理解：权限系统像门禁：模型可以提出想进哪扇门，但开不开门由宿主程序决定。
工程重点：规则要可解释、可审计、可配置；默认收紧高风险能力，低风险读操作才适合自动放行。
权限闸门：工具调用先分类，再执行
工具请求read_file src/app.ts
ALLOW低风险、命中白名单
直接执行结果写回上下文
工具请求git push / rm / 外部写入
ASK USER有副作用或不可逆
等待确认用户同意后才继续
工具请求越权路径 / 敏感数据
DENY违反边界
阻断并解释不进入 handler

权限判断属于系统责任，不能让模型自己决定「这次应该安全」。

越能干的 Agent 越需要权限边界16
第二部分 Agent 开发入门 / s08

Hook 系统：在关键生命周期插入扩展

Hook 把日志、审计、格式化、通知等横切能力放到生命周期节点上，而不是塞进主循环。

要解决的问题：Agent 主循环会越来越重：既要运行工具，又要记录日志、做审计、发通知、补权限。
核心机制：在 SessionStart、PreToolUse、PostToolUse、Stop 等阶段触发 hook。
可以这样理解：Hook 像流程里的插座：核心电路不变，需要扩展时把插件接上去。
工程重点：hook 输入输出要稳定；失败要隔离；慢 hook 要超时；不能因为扩展失败拖垮主流程。
生命周期 Hook：扩展挂在节点上
SessionStart加载规则、初始化日志、准备环境。
UserPromptSubmit记录输入、补充上下文、做前置检查。
PreToolUse权限补丁、参数检查、审计记录。
PostToolUse截断输出、格式化结果、写入追踪。
Stop收尾通知、保存状态、生成摘要。

讲解重点：Hook 的价值是让核心循环保持干净，同时给团队留下稳定的扩展点。

Hook 让核心循环保持干净17
第二部分 Agent 开发入门 / s09

记忆系统：只记未来仍有价值的东西

记忆不是聊天记录仓库，而是把未来还会用到的事实、偏好和项目规则保存下来。

要解决的问题：跨会话后，Agent 会忘记项目怎么跑、用户偏好、团队约定和上次验证过的结论。
核心机制：区分短期状态、长期记忆、项目规则、原始记录；按需要检索，而不是全量塞回提示词。
可以这样理解：记忆像工程手册，不是监控录像；要写能指导下一次行动的信息。
工程重点：记忆要有来源和时间；能更新、能冲突处理；当前文件和当前验证永远优先于旧记忆。
记忆分层：不同信息放不同位置
短期状态当前任务、当前计划、正在处理的文件。会话结束后不一定保留。
项目规则测试命令、目录约定、代码风格、部署入口。下次任务仍然有用。
用户偏好输出习惯、常用流程、明确说过要记住的默认值。
原始记录完整日志、历史对话、工具输出。用于追溯，不默认塞进上下文。

好记忆要能回答「下次遇到同类任务，它能少走哪一步弯路？」

记忆不是聊天记录仓库18
第二部分 Agent 开发入门 / s10

系统提示词：用拼装流水线管理规则

Prompt 不是一整坨文本，而是由多层规则按优先级拼装出来的运行配置。

要解决的问题：系统提示词会越来越长：平台规则、项目规则、工具说明、Skill、当前任务约束混在一起难维护。
核心机制：按来源拼装：基础行为规则、项目指令、工具目录、Skill 摘要、权限模式、当前任务 brief。
可以这样理解：Prompt 像应用启动配置，不应该手写一份巨大字符串，而应该有稳定的装配顺序。
工程重点：冲突要有优先级；动态内容不要污染长期规则；每一层来源都要可追踪。
Prompt 组装：稳定顺序 + 清晰来源
1
基础 Agent 行为规则
最高优先级
2
项目 AGENTS.md / 团队约束
本仓库规则
3
工具目录 / Skill 摘要 / 权限模式
运行时能力
4
当前任务目标、计划、已知约束
本轮动态

讲解重点：提示词工程到工程化阶段，核心不是写一句神奇 prompt，而是管理规则来源和优先级。

Prompt 是运行配置，不是一坨文本19
第二部分 Agent 开发入门 / s11

错误恢复：失败要能被理解和重试

好 Agent 不是不失败，而是能把失败转成可理解的状态，并选择下一步恢复动作。

要解决的问题：命令失败、路径不存在、网络超时、权限拒绝都会打断任务；整屏 stderr 会让模型更混乱。
核心机制：把错误分类成可重试、需改参数、需用户输入、不可恢复，并给出下一步建议。
可以这样理解：错误恢复像客服分诊：先判断病因，再决定重试、换方案、问用户，还是明确停止。
工程重点：返回短摘要、退出码、关键错误片段、已尝试动作和建议动作；避免把噪音原封不动塞回上下文。
错误恢复矩阵：先分类，再行动
可重试网络抖动、临时超时。动作：退避重试，限制次数。
需改参数路径错误、命令参数错。动作：读错误摘要，修正后再跑。
需用户输入缺凭证、权限确认、需求不清。动作：提出具体问题。
不可恢复策略禁止、资源不存在、目标冲突。动作：停止并说明原因。

失败信息要「可行动」，模型需要的是下一步线索，不是一整屏未经整理的报错。

好 Agent 不是不失败，而是会恢复20
第二部分 Agent 开发入门 / s12

任务系统：把大目标拆成持久工作图

Todo 管当前注意力，任务系统管跨会话、跨角色、可恢复的长期工作。

要解决的问题：复杂目标不是一轮对话能完成的，必须保存任务状态、依赖关系、负责人、产物和日志。
核心机制：把大目标拆成 task graph；每个 task 有 pending、ready、running、blocked、done 等状态。
可以这样理解：任务系统像项目看板，不只是模型脑子里的计划，而是所有 Agent 都能读写的共同现实。
工程重点：状态变更要落盘；依赖解除后才能 ready；产物和日志要可追溯，方便中断后继续。
任务图：长期工作的共同状态
task-01 需求澄清done · artifact: brief.md
task-02 实现功能ready · owner: coder · depends on 01
task-03 联调验证blocked · 等待测试环境
task-04 交付总结pending · depends on 02 + 03

任务系统把「下一步」从聊天上下文里拿出来，变成可恢复、可交接、可审计的状态。

任务板是多轮工作的共同现实21
第二部分 Agent 开发入门 / s13

后台任务：慢操作不阻塞主循环

后台任务让慢操作脱离主循环等待，但结果仍要回到同一套任务和消息系统。

要解决的问题：跑测试、构建、爬数据、批量处理都可能很久；如果同步等待，Agent 主循环会被卡住。
核心机制：background_run 立即返回 task_id；慢执行写日志；完成后通过通知队列把摘要送回主循环。
可以这样理解：不是复制一个主脑并行思考，而是把耗时动作放到后台执行槽位。
工程重点：后台任务要能查询状态、取消、超时；通知只放摘要，完整输出放文件或 artifact。
后台执行流：快返回，慢执行，完成再通知
主循环发起run tests / build / crawl
→
立即返回 task_id主循环继续规划或处理别的事
后台执行槽写完整日志和产物
→
通知队列done / failed / timeout 摘要
主循环读取通知决定重试、修复或继续
→
任务状态更新running → done / blocked

讲解重点：后台任务不是脱离系统单独跑，最终结果必须回到任务状态和上下文里。

并行的是执行槽位，不是复制主脑22
第二部分 Agent 开发入门 / s14

定时调度：把未来任务放回 Agent Loop

调度器只负责在未来唤醒 Agent，真正思考和执行仍然进入 Agent Loop。

要解决的问题：提醒、巡检、日报、周报、监控回访不能依赖当前会话一直开着。
核心机制：保存 schedule + prompt；到时间后把 prompt 作为新输入追加回消息系统。
可以这样理解：定时器像闹钟，它只负责叫醒；醒来之后怎么做，仍由 Agent 根据上下文、权限和工具完成。
工程重点：触发要可追踪、可暂停、可更新；执行结果仍走权限、工具、任务状态和错误恢复。
调度唤醒：未来输入回到同一个 Loop
创建计划每天 09:00 / 每周一 / 30 分钟后
→
保存 prompt任务内容自包含，可独立唤醒
到点触发scheduler due
→
追加新输入进入 messages / task queue
Agent Loop 执行权限、工具、记忆、错误恢复
→
产出结果通知用户或更新任务

调度能力的关键是「回到统一运行时」，而不是在定时器里另写一套逻辑。

定时器只负责唤醒，不负责思考23
第二部分 Agent 开发入门 / s15

Agent 团队：让角色长期存在

多 Agent 不是一次性 subagent，而是让不同角色长期存在、拥有职责和状态。

要解决的问题：一个 Agent 很难同时做好规划、探索、实现、测试和交付，还保持所有上下文都清晰。
核心机制：每个角色有自己的职责、工具范围、上下文、记忆和任务队列。
可以这样理解：团队 Agent 像长期队友，不是临时函数调用；每个人知道自己负责什么，也知道怎么交接。
工程重点：角色边界要清楚，输入输出要结构化；否则团队只会增加沟通噪音。
Agent 团队：职责、工具、状态分开

Lead Agent

拆目标和优先级
分派任务和验收
整合最终交付

长期角色

Explorer：查资料、扫代码、找证据
Coder：实现改动、维护 diff
Tester：运行验证、复现失败
Writer：整理说明、产出文档

讲解重点：团队化的价值不是「人多」，而是把不同上下文和工具边界隔离开。

队友不是一次性函数调用24
第二部分 Agent 开发入门 / s16

团队协议：协作要有可追踪规则

多 Agent 要靠协议协作，而不是靠自然语言随口喊话。

要解决的问题：多个 Agent 同时工作会抢任务、覆盖文件、丢交接、重复验证、互相不知道状态。
核心机制：任务认领、状态更新、消息、交接、退出都走结构化协议。
可以这样理解：协议像团队协作制度：谁认领、谁负责、什么时候交接，都要写到系统里。
工程重点：消息要能 ack；任务要能锁定；退出前要交接；领导可以请求停止，但不能直接抹掉执行者手上的状态。
团队协议：把协作动作结构化
Claim原子认领 ready task
Updaterunning / blocked / done
Ack收到请求，确认责任
Handoff产物、风险、下一步
Exit先交接，再退出
Audit消息、状态、产物留痕

没有协议的多 Agent 只是多条聊天；有协议才是可追踪的协作系统。

多 Agent 需要协议，不只需要热闹25
第二部分 Agent 开发入门 / s17

自主代理：空闲时自己找活干

自治不是随便发挥，而是在任务板和协议约束下，从 idle 状态主动认领 ready task。

要解决的问题：长期团队不能靠 Lead 每一步手动叫人；空闲角色应该能自己发现可做工作。
核心机制：IDLE 时轮询 inbox 和 task board；发现 ready task 后原子认领，进入 working。
可以这样理解：自主代理像有规则的值班同事：先看有没有消息，再看有没有可认领任务。
工程重点：认领要加锁，避免重复工作；没有 ready task 就保持 idle，不要自己编任务。
自治循环：只在规则下主动找活
IDLE没有当前任务，等待消息或任务板变化。
Check Inbox处理点名请求、交接消息、停止请求。
Claim Ready Task原子认领，写 owner 和 running 状态。
Work + Report执行、产出 artifact、更新任务状态。
No Ready Task继续 idle，不凭空扩展目标。
Back To Board完成后回到任务板找下一项。

讲解重点：自治的边界来自任务系统和团队协议，不是让 Agent 无限自由发挥。

自治是在规则下认领 ready task26
第二部分 Agent 开发入门 / s18

Worktree 隔离：队友各自在独立工作区动手

多 Agent 真正同时改代码时，需要用 worktree / branch 把工作区隔离开。

要解决的问题：多个 Agent 在同一个目录里改文件，很容易覆盖、误删、混入彼此未完成的改动。
核心机制：每个任务或角色使用独立 worktree 和 branch；完成后通过 diff、测试、合并流程汇总。
可以这样理解：Worktree 像给每个队友一条独立车道，大家并行开车，最后在合并口检查汇入。
工程重点：分配写入范围；合并前看 diff 和测试；遇到冲突人工或主代理裁决，不自动吞掉别人的修改。
Worktree 车道：并行动手，集中汇总
main稳定基线
task-024 worktreeCoder 修改 UI
task-025 worktreeTester 补验证
隔离修改互不覆盖工作区
各自测试本车道先跑通
提交产物diff + logs + notes
汇总口主代理评审
处理冲突明确谁覆盖谁
合并回主线最终测试和交付

Worktree 降低冲突概率，但不能代替合并前的测试和代码审查。

隔离降低冲突，汇总需要校验27
第二部分 Agent 开发入门 / s19

MCP 与插件：把外部能力标准化接入

平台化以后，Agent 不应该为每个外部系统单独硬编码，而要通过 MCP / 插件标准化接入能力。

要解决的问题：浏览器、数据库、Wiki、GitHub、内部系统、桌面应用都有不同协议和鉴权方式。
核心机制：MCP / 插件把 resources、tools、prompts 统一暴露给 Agent，宿主负责注册和权限管理。
可以这样理解：MCP 像 Agent 的 USB-C 接口：外部系统各不相同，但接入形态尽量标准化。
工程重点：工具描述要清楚；权限最小化；敏感数据不随意传出；失败和速率限制要包装成可恢复结果。
MCP / 插件：外部能力接入层
外部系统Wiki / GitHub / DB / Browser
→
MCP Server / Plugin封装认证、协议、数据格式
统一能力resources / tools / prompts
→
Agent Host注册、权限、调用、结果回写
模型可见清晰描述 + schema
→
安全执行最小权限、审计、错误恢复

MCP 解决的是生态接入问题，让 Agent 能用更多系统，但仍受同一套权限和上下文规则约束。

平台化以后，能力来自生态28
